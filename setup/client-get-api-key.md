# Как клиенту получить API-ключ

API-ключ создает сам клиент в личном кабинете. Администратор сначала должен разрешить клиенту API-доступ, а клиент уже создает ключ, сохраняет Bearer-токен и HMAC-секрет.

Эта инструкция описывает путь по интерфейсу, который видит клиент.

## 1. Открыть меню профиля

Войдите в аккаунт обменника.

В правой верхней части сайта нажмите на иконку профиля. Откроется меню аккаунта.

В меню выберите:

```text
Настройки
```

В этом же меню могут быть “Мои заявки”, “Мои реквизиты”, “Скидки и промокоды”, “Реферальная программа” и другие разделы. Для API нужен именно раздел “Настройки”.

## 2. Открыть вкладку API

На странице настроек откройте вкладку:

```text
API
```

На этой вкладке находится блок:

```text
API
Управляйте ключами публичного API и HMAC-подписью запросов.
```

Ниже отображается счетчик ключей:

```text
API-ключи
Использовано 9 / 10
```

или, если лимит уже достигнут:

```text
Использовано 10 / 10
Достигнут лимит API-ключей. Максимум ключей на аккаунт: 10.
```

Если лимит достигнут, новый ключ создать нельзя. Удалите старый ключ или попросите администратора увеличить лимит.

## 3. Нажать “Создать ключ”

Нажмите кнопку:

```text
Создать ключ
```

После создания появится блок:

```text
Новый ключ создан
Сохраните Bearer-токен и HMAC-секрет сейчас. После обновления страницы останутся только безопасные данные ключа.
```

Это самый важный момент. Bearer-токен и HMAC-секрет показываются только один раз.

## 4. Скопировать Bearer-токен

В блоке “Новый ключ создан” скопируйте:

```text
Bearer-токен
```

Bearer-токен используется в каждом приватном API-запросе:

```http
Authorization: Bearer YOUR_API_KEY
```

Пример:

```bash
curl -sS https://example.com/api/v3/private/health/client \
  -H "Accept: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## 5. Скопировать HMAC-секрет

В том же блоке скопируйте:

```text
HMAC-секрет
```

Он нужен для подписи запросов. Алгоритм отображается ниже, например:

```text
Алгоритм подписи: hmac-sha256-v1
```

Если ключ помечен как `HMAC включён`, разработчик должен отправлять HMAC headers:

```http
X-Api-Timestamp: 1782190000
X-Api-Nonce: req-20260623-001
X-Api-Signature: sha256=...
```

Подробно: [HMAC-подпись запросов](../architecture/hmac-signature.md).

## 6. Проверить карточку ключа

После создания ключ появится в списке. В карточке ключа обычно видны:

| Поле | Что означает |
| --- | --- |
| Название ключа | Понятное имя ключа, например `production-crm`. |
| ID ключа | Внутренний номер для поиска и поддержки. |
| Белый список IP | IP-адреса, с которых разрешены запросы. |
| Ключ включён | Тумблер активности ключа. |
| Создан | Дата создания ключа. |
| Последнее использование | Когда ключ последний раз применялся. |
| Scope-ы | Какие действия разрешены ключу. |
| `HMAC включён` | Запросы должны подписываться HMAC. |
| `Активен` | Ключ включен и может работать. |
| `Удалить` | Удаляет ключ. |

## 7. Настроить белый список IP

Если интеграция работает с backend-сервера, в поле “Белый список IP” можно указать разрешенные IP:

```text
127.0.0.1
192.168.1.0/24
203.0.113.10
```

Для production нужно указать публичный IP сервера, с которого приложение отправляет запросы в API. Если запросы идут через NAT, proxy или load balancer, нужен именно внешний IP.

Если IP allow-list включен, запросы с других IP будут получать ошибку доступа.

## 8. Включить или выключить ключ

В карточке есть тумблер:

```text
Ключ включён
```

Если выключить ключ, приватные API-запросы с этим Bearer-токеном перестанут работать. Это удобно, если нужно временно остановить интеграцию без удаления ключа.

После изменения названия, IP или активности нажмите:

```text
Сохранить ключ
```

## 9. Передать данные разработчику

Разработчику нужны:

```text
Base URL: https://example.com/api/v3
Bearer-токен: ...
HMAC-секрет: ...
Алгоритм: hmac-sha256-v1
Список scopes
Нужен ли IP allow-list
```

Передавайте эти данные только через безопасный канал: password manager, vault, зашифрованное сообщение или временную защищенную ссылку.

Нельзя отправлять ключи в открытом виде в email, чат, GitHub issue, скриншотах или задачах.

## 10. Если секреты не сохранили

Если страница обновилась, а Bearer-токен или HMAC-секрет не были сохранены, посмотреть их повторно нельзя.

Что делать:

1. Удалить старый ключ или выключить его.
2. Создать новый ключ.
3. Сразу скопировать Bearer-токен и HMAC-секрет.
4. Передать новые данные разработчику.
5. Проверить `GET /private/health/client`.

## 11. Когда удалять ключ

Удалите ключ, если:

- интеграция больше не используется;
- секрет случайно попал в переписку или лог;
- подрядчик больше не должен иметь доступ;
- создан новый ключ после ротации;
- лимит `10 / 10` достигнут и нужно освободить место.
