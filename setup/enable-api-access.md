# Enable API access

API-доступ включается администратором, а ключ создает клиент в своем кабинете. Это разделяет ответственность: администратор задает правила, клиент управляет секретами, разработчик подключает интеграцию.

## Роли

| Роль | Ответственность |
| --- | --- |
| Администратор обменника | Включает API, задает лимиты, scopes, HMAC policy, IP policy и webhook limits. |
| Владелец клиентского аккаунта | Создает ключ, сохраняет секреты, ротирует и отзывает ключи. |
| Разработчик интеграции | Отправляет запросы, подписывает их, обрабатывает ошибки и webhooks. |
| Support | Помогает по request ID, статусам заявок, лимитам и ошибкам доступа. |

## Настройка администратором

В админке откройте раздел API и проверьте:

1. API включен глобально.
2. Клиенту разрешен API-доступ.
3. Для клиента или ключа назначены нужные scopes.
4. Установлены rate limits и quotas.
5. Настроена HMAC policy.
6. При необходимости включен IP allow-list.
7. Настроен лимит webhook endpoints.

Рекомендуемая production-политика:

| Настройка | Рекомендация |
| --- | --- |
| API access | Включать только клиентам, которым реально нужна интеграция. |
| Scopes | Выдавать минимальный набор прав. |
| HMAC | Требовать минимум для write-запросов. |
| IP allow-list | Включать для серверных production-интеграций. |
| Quotas | Ставить дневной и месячный лимит, соответствующий бизнес-нагрузке. |
| Admin accounts | Не использовать админские аккаунты для публичного API. |

## Настройка клиентом

Клиент в личном кабинете:

1. Открывает раздел API-ключей.
2. Создает ключ с понятным названием, например `production-crm`.
3. Сохраняет Bearer token и HMAC secret сразу после создания.
4. Передает секреты разработчику безопасным способом.
5. После запуска удаляет тестовые ключи.

Секреты показываются только при создании или ротации. Если секрет потерян, создайте новый ключ или выполните ротацию.
